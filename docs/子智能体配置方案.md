# Codex Desktop 第三方子智能体配置方案

## 目标

Codex Desktop 主会话继续使用 `GPT-5.5`。仅在用户明确要求委派时使用以下自定义子智能体：

| 子智能体 | Provider | 实际模型 | 适用任务 | 多模态输入 |
|---|---|---|---|---|
| `deepseek-v4-pro` | `cf_deepseek_adapter` | `deepseek-v4-pro` | 复杂编码、调试和实现 | 不支持 |
| `deepseek-v4-flash` | `cf_deepseek_adapter` | `deepseek-v4-flash` | 快速探索、总结和聚焦修改 | 不支持 |
| `minimax-m3` | `minimax_m3` | `MiniMax-M3` | 长上下文、工具调用和多模态任务 | 官方 API 支持 |

Codex 只在用户明确要求子智能体或并行工作时创建子智能体，不根据描述自动委派。

## 路由设计

两个 DeepSeek 子智能体都向本项目公开的 `deepseek-codex` 模型别名发送请求。Worker 根据
`reasoning.effort` 选择上游模型：

- `none` -> `UPSTREAM_TEXT_MODEL=deepseek-v4-flash`，关闭 thinking；
- `high` -> `UPSTREAM_REASONING_MODEL=deepseek-v4-pro`，开启 thinking。

MiniMax-M3 原生提供 Responses API。当前 Key 属于 MiniMax 国内站，因此使用
`POST https://api.minimaxi.com/v1/responses`，由 Codex 直接调用，
不经过本项目 Worker。初始配置使用 `model_reasoning_effort = "none"`，避免 MiniMax 的明文
reasoning item 与 Codex 的 encrypted reasoning 扩展产生未经验证的兼容问题。

## 认证方案一：macOS Keychain

macOS 推荐通过 Codex command-backed auth 从 Keychain 获取 Bearer Token。长期 Token 不写入
`config.toml`、shell profile 或仓库。

| Keychain service | 保存内容 |
|---|---|
| `codex-deepseek-adapter` | 本项目 Worker 的 `ADAPTER_BEARER_TOKEN` |
| `codex-minimax-m3` | MiniMax API Key 或 Subscription Key |

### 保存 DeepSeek Adapter Token

`codex-deepseek-adapter` 的值必须与 Worker 的 `ADAPTER_BEARER_TOKEN` 完全一致。部署者可在
项目目录中一次性写入 Keychain 和 Cloudflare：

```zsh
read -rs "ADAPTER_TOKEN?请输入 ADAPTER_BEARER_TOKEN: "
echo
security add-generic-password \
  -U \
  -a "$(id -un)" \
  -s "codex-deepseek-adapter" \
  -l "Codex DeepSeek Adapter Bearer Token" \
  -w "$ADAPTER_TOKEN"
printf '%s' "$ADAPTER_TOKEN" | npx wrangler secret put ADAPTER_BEARER_TOKEN
unset ADAPTER_TOKEN
```

只接入已有 Worker 时，不执行 `wrangler secret put`，由 Worker 管理者通过安全渠道提供当前
Token 后写入 Keychain。

### 保存 MiniMax Key

```zsh
read -rs "MINIMAX_KEY?请输入 MiniMax API Key: "
echo
security add-generic-password \
  -U \
  -a "libo" \
  -s "codex-minimax-m3" \
  -l "Codex MiniMax M3 API Key" \
  -w "$MINIMAX_KEY"
unset MINIMAX_KEY
```

`read -s` 隐藏输入，Key 不会进入 shell 历史。不要使用 `-A`。

### Provider 配置

```toml
[model_providers.cf_deepseek_adapter]
name = "Cloudflare DeepSeek Responses Adapter"
base_url = "https://codex.bobocai.win/v1"
wire_api = "responses"
request_max_retries = 0
stream_max_retries = 0
supports_websockets = false

[model_providers.cf_deepseek_adapter.auth]
command = "/usr/bin/security"
args = ["find-generic-password", "-a", "<macOS 用户名>", "-s", "codex-deepseek-adapter", "-w"]
timeout_ms = 5000
refresh_interval_ms = 0

[model_providers.minimax_m3]
name = "MiniMax-M3 Responses API"
base_url = "https://api.minimaxi.com/v1"
wire_api = "responses"
request_max_retries = 0
stream_max_retries = 0
supports_websockets = false

[model_providers.minimax_m3.auth]
command = "/usr/bin/security"
args = ["find-generic-password", "-a", "<macOS 用户名>", "-s", "codex-minimax-m3", "-w"]
timeout_ms = 5000
refresh_interval_ms = 0
```

将 `<macOS 用户名>` 替换为 `id -un` 的输出。录入后可验证长度非零，但不要输出 Token 本身：

```zsh
TOKEN="$(security find-generic-password -a "$(id -un)" -s "codex-deepseek-adapter" -w)"
echo "${#TOKEN}"
unset TOKEN
```

## 认证方案二：跨平台环境文件

不使用 Keychain 时，通过 provider 的 `env_key` 读取环境变量。此方法适用于 macOS、Linux
和 Windows。删除对应的 `[model_providers.<id>.auth]` 表，并在 provider 中配置：

```toml
[model_providers.cf_deepseek_adapter]
name = "Cloudflare DeepSeek Responses Adapter"
base_url = "https://codex.bobocai.win/v1"
env_key = "CODEX_GATEWAY_API_KEY"
wire_api = "responses"
request_max_retries = 0
stream_max_retries = 0
supports_websockets = false

[model_providers.minimax_m3]
name = "MiniMax-M3 Responses API"
base_url = "https://api.minimaxi.com/v1"
env_key = "MINIMAX_API_KEY"
wire_api = "responses"
request_max_retries = 0
stream_max_retries = 0
supports_websockets = false
```

`auth` 和 `env_key` 不能同时存在。Codex Desktop 可能不继承 shell 环境，因此将变量写入
`CODEX_HOME/.env`，修改后完全退出并重启 Codex Desktop。

### macOS 与 Linux

默认路径是 `~/.codex/.env`：

```sh
umask 077
mkdir -p ~/.codex
touch ~/.codex/.env
chmod 600 ~/.codex/.env
${EDITOR:-vi} ~/.codex/.env
```

在编辑器中写入：

```dotenv
CODEX_GATEWAY_API_KEY=<adapter-bearer-token>
MINIMAX_API_KEY=<minimax-api-key>
```

将占位值替换为真实 Token。该文件是权限受限的明文文件，不得提交到仓库或同步到公共网盘。

### Windows PowerShell

默认路径是 `%USERPROFILE%\.codex\.env`：

```powershell
$codexHome = Join-Path $env:USERPROFILE ".codex"
$envFile = Join-Path $codexHome ".env"
New-Item -ItemType Directory -Force $codexHome | Out-Null
@"
CODEX_GATEWAY_API_KEY=<adapter-bearer-token>
MINIMAX_API_KEY=<minimax-api-key>
"@ | Set-Content -Encoding ascii $envFile
icacls $envFile /inheritance:r /grant:r "${env:USERNAME}:(R,W)"
```

Windows ACL 命令移除继承权限，仅向当前用户授予读写权限。仍应避免将 `.env` 放入仓库。

## 配置文件

- Provider 和并发限制：`~/.codex/config.toml`
- DeepSeek Pro：`~/.codex/agents/deepseek-v4-pro.toml`
- DeepSeek Flash：`~/.codex/agents/deepseek-v4-flash.toml`
- MiniMax M3：`~/.codex/agents/minimax-m3.toml`

Agent 文件中的 `model_provider` 和 `model` 覆盖父会话；未显式设置的权限和 sandbox 配置
继续继承父会话。两个 DeepSeek Agent 的描述与指令均明确禁止图片、音频、视频及其他多模态输入。

## 使用方式

明确指定一个角色：

```text
请创建 deepseek-v4-flash 子智能体，快速探索这个仓库的认证流程并返回文件引用。
```

```text
请创建 deepseek-v4-pro 子智能体，修复这个并发问题并运行相关测试。
```

```text
请创建 minimax-m3 子智能体，结合这张截图和仓库代码分析 UI 状态异常。
```

并行委派：

```text
并行创建 deepseek-v4-flash 和 minimax-m3 两个子智能体：前者分析代码路径，后者分析截图；
等待两者完成后汇总结论。
```

## 验证与限制

1. 修改 Agent 或 Provider 配置后，完全退出并重新启动 Codex Desktop。
2. DeepSeek Pro 是否实际路由到 Pro 由 Worker 的 `UPSTREAM_REASONING_MODEL` 决定，必须部署本方案对应的 Worker 配置。
3. MiniMax 官方 Responses schema 支持流式输出、function tools、连续工具回合、reasoning、图片和视频输入；正式使用前仍应以真实 API Key 完成 Codex 文本、工具和多模态 smoke。
4. 子智能体不会出现在主会话模型选择菜单中；它们通过明确的委派指令使用。
5. 当前 MiniMax Key 已在国内端点通过认证，但返回 `2056` Token Plan 用量上限；恢复额度后才能完成真实 smoke。
