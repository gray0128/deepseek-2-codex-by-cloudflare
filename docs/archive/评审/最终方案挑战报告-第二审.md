# 对《DeepSeek 接入 Codex：最终实施方案》的第二轮挑战评审

---
评审日期：2026-06-19
评审对象：[`docs/最终方案.md`](../../最终方案.md)
评审立场：与 [第一轮挑战报告](./最终方案挑战报告.md) 正交。第一轮聚焦"方案内部未冻结的设计"（response id 编码、SQL 事务边界、SSE golden、SLO 数值等）；本轮聚焦"方案与项目定位、平台硬限制、第三方能力假设、替代架构之间的张力"。
前提声明：本轮不重复第一轮已论证的点，仅在交叉处引用。本轮部分结论通过查证 Codex 与 DeepSeek 官方文档得出，已标注来源。

## 0. 结论

一句话：方案在"正确性优先"方向上已经走得足够远，但越走越像一个中型生产系统，而 `AGENTS.md` 把本项目定义为"internal small project，prioritize efficiency, stability, delivery speed"。这两者之间的张力贯穿全篇，且未被方案正面回应。

**如果"内部小项目"为真**，方案中至少有三层设计是过度投资，应删减后再开工；**如果"内部小项目"为假**（实际面向多用户或公开服务），方案的运维、安全、SLO 章节又远远不够。**项目定位不清是当前最大风险，比任何技术空白都更优先。**

本轮新增的关键查证：

| 查证项 | 结论 | 对方案的影响 |
|---|---|---|
| Codex `wire_api` 是否可选 `chat_completions` | 否，仅支持 `responses`（[config-reference](https://developers.openai.com/codex/config-reference/)） | 协议转换层是"硬性税"，无法回避；方案的必要性被加强，但这也意味着 Phase -1 必须先验证最小可行路径，因为没有绕路 |
| DeepSeek `deepseek-v4-pro` 模型名是否真实 | 是，真实存在（[DeepSeek API Docs](https://api-docs.deepseek.com/)） | 方案配置示例可成立 |
| `deepseek-chat`/`deepseek-reasoner` 旧名状态 | 将于 2026/07/24 UTC 弃用 | 方案若在任何子文档引用旧名将有时效风险 |
| DeepSeek thinking 与非 thinking 是否同一模型名 | 文档显示当前为 `deepseek-v4-pro` 与 `deepseek-v4-flash` 两个模型名 | **方案"单一模型名覆盖所有 effort 级别"的假设存疑**，见 §3.2 |

## 1. 项目定位与方案复杂度的张力（最高优先级）

### 1.1 体量落差

`AGENTS.md` 开篇："This project is an internal small project. Prioritize efficiency, stability, and delivery speed."

而 `最终方案.md` 实际包含：四阶段实施（Phase -1 到 3）、Durable Object + SQLite 状态层、完整 Responses SSE 契约、reasoning 分阶段准入、幂等与 request id 协议、SLO 矩阵、上线门槛清单、密钥轮换、TTL 清理、Gateway 对比……这已经不是"小项目"的体量，而是"对外 SaaS 适配层"的规格。

第一轮挑战默认接受了这个体量并要求补全细节；本轮要质疑的是体量本身。

### 1.2 三处可能过度投资的设计

| 设计 | 过度投资的理由 | 更轻的替代 |
|---|---|---|
| Durable Object + SQLite 完整状态层 | 若 Phase -1 证明 Codex 每次发全量 input，DO 仅用于 reasoning_content 与 call id 映射 | 用 Workers KV / D1 / 甚至内存 + 短 TTL 起步，证明必要后再升 DO |
| 完整 Responses SSE 事件生命周期 + golden fixture 双套 | 第一版只需"Codex 单句文本能跑通"即可验证路线 | 先手写最小事件序列跑通 Codex，再补契约 |
| 四阶段 + SLO + 上线门槛矩阵 | 内部小项目通常 1 个使用者、低并发 | 砍掉 Phase 3 Gateway 与并发 SLO，改为"够用即止" |

### 1.3 若定位为"个人/内部小项目"应砍什么

- §3.1 的"第一版最终目标"砍到：`POST /v1/responses`（stream=true，纯文本）+ `GET /v1/models` + `/healthz`。tools、thinking、previous_response_id 全部推到 v2。
- §4.2 的 DO 状态层先用 Workers KV 存 `reasoning_content`（TTL 1 小时），证明不够再升 DO。
- §5 的 Phase 0 即可交付"Codex 能对话"，Phase 1 再加工具。Phase 2/3 暂不排期。
- §6 的"2 并发与 6 并发 SLO"删除，改为"单用户连续 30 轮无 500"。

**挑战结论：** 方案应在开头显式回答"这是给谁用的"。如果是作者本人用，§1.2 的删减应立即执行；如果计划共享，则 §6 的安全/隐私/多租户章节必须补全——当前方案两头不靠。

## 2. 根本前提已被查证：协议转换层确属硬性税

### 2.1 查证结果

本轮通过 [Codex config-reference](https://developers.openai.com/codex/config-reference/) 确认：自定义 `model_provider` 的 `wire_api` **仅支持 `responses`**，不支持 `chat_completions`。这意味着：

- 方案 §2.1"协议转换层不可省略"的判断**成立且无可回避**；
- 第一轮未质疑这一点，本轮质疑后反而**加强**了方案的必要性；
- 不存在"让 Codex 直接说 Chat Completions"的绕路。

### 2.2 但这带来一个被低估的后果

既然协议转换是硬性税，那么 **Phase -1 的首要目标不是采 fixture，而是先用最小代码证明"Responses → Chat Completions → DeepSeek"这条链路在 Codex 里能跑通一个字**。理由：

- 如果链路本身有未知的硬阻塞（如 Codex 要求某个 Responses 字段，DeepSeek 无法提供），越早暴露越好；
- fixture 采集是"把链路做对"的输入，但"链路是否通"是更前置的问题；
- 当前仓库无任何实现，Phase -1 描述偏向文档交付，未要求"跑通一个端到端 hello world"。

**挑战结论：** 在 Phase -1 的交付物中强制加入一项：`examples/hello-codex.md` —— 用最简 Worker（无 DO、无工具、无 thinking）让 Codex 完成一次纯文本对话并截图/日志。这一项跑不通，所有后续设计都是空中楼阁。

## 3. 对 DeepSeek API 能力的"文档级假设"未做端点验证

### 3.1 模型名版本锁定缺失

DeepSeek 正处于模型名迁移期：

- `deepseek-chat` / `deepseek-reasoner` 将于 **2026/07/24 UTC** 弃用（[DeepSeek API Docs](https://api-docs.deepseek.com/)）；
- 当前正式名为 `deepseek-v4-pro` 与 `deepseek-v4-flash`。

方案 §8 配置示例用 `deepseek-v4-pro`，正确。但方案未规定：

- 是否锁定到具体版本号（如带日期 tag）；
- DeepSeek 静默升级模型行为时如何发现；
- 服务端配置的模型名与 Codex `model` 字段的映射规则（Codex 发来的 `model` 是 `deepseek-v4-pro` 还是别的别名？）。

**挑战：** 应在 §4.1 upstream client 模块增加"模型名白名单 + 别名映射"，并在 `GET /v1/models` 返回当前可用的精确名，避免 Codex 配置漂移。

### 3.2 单一模型名覆盖所有 effort 级别的假设存疑（硬挑战）

§4.4 的 effort 映射表对 `none`/`low`/`medium`/`high`/`xhigh` 全部使用同一模型名（隐含 `deepseek-v4-pro`），仅切换 `thinking.type` 与 `reasoning_effort`。但 DeepSeek 文档显示存在 `deepseek-v4-flash`（非 thinking 取向）与 `deepseek-v4-pro`（thinking 取向）两个模型名。

这带来两个未回答的问题：

1. `deepseek-v4-pro` 是否支持 `thinking.type=disabled`？若不支持，`effort=none` 必须路由到 `deepseek-v4-flash`，方案需要 **effort → model 映射表**，而非单一模型名。
2. 若 `v4-pro` 与 `v4-flash` 的工具调用行为、上下文窗口、定价不同，混用同一 thread 会出现历史不兼容或计费跳变。

**挑战结论：** §4.4 必须增加"模型选择策略"子节，明确：

- 每个 effort 级别路由到哪个 DeepSeek 模型名；
- 同一 conversation 是否允许跨模型名续接（parent 链跨模型时的 reasoning_content 兼容性）；
- 服务端配置如何覆盖客户端 `model` 字段。

### 3.3 parallel_tool_calls 与 tool_choice 的真实支持未验证

方案 §3.1 对 `parallel_tool_calls` 采取"验证后透传，否则拒绝"。但"验证"的对象是 DeepSeek 端点，不是 Codex。DeepSeek Tool Calls 文档（[链接](https://api-docs.deepseek.com/guides/tool_calls)）是否明确支持：

- `tool_choice: "required"` / `"auto"` / 指定函数名；
- `parallel_tool_calls: true` 且实际返回多个 tool_calls；
- thinking 模式下 tool_calls 与 reasoning_content 的交错顺序。

这些方案均未引用 DeepSeek 文档原文。**挑战：** Phase -1 的 fixture 采集应同步包含"对 DeepSeek 端点的直接探测脚本"，输出一份 `docs/deepseek-capabilities.md`，否则 §3.1 的"验证后透传"没有验证对象。

## 4. Cloudflare Workers 平台硬限制未被列出

方案通篇未提及平台限制，而这些限制会直接塑造 §4 的设计。

### 4.1 subrequest 限制与取消传播

- Workers 付费版 subrequest 上限为 **1000**（免费版 50）。一个 Responses 请求至少 1 个 upstream fetch；但若未来加入 Gateway、密钥轮换的 Secrets 读取、DO 调用，subrequest 计数会上升。
- 取消传播依赖 `fetch` 的 `AbortSignal` + `ReadableStream.cancel()`，Workers 支持但行为与 Node.js 有差异（`abort` 后上游连接是否真正关闭需验证）。
- §4.5 的"客户端断开立即取消 reader 和 upstream fetch"在 Workers 上需要显式 `request.signal.addEventListener('abort', ...)`，不能假设 GC 会处理。

**挑战：** §4.5 应增加平台特定的取消实现说明，并增加一项测试：客户端断开后用 `wrangler tail` 确认上游 fetch 在 N 秒内 abort。

### 4.2 CPU 时间与流式

- Workers 单请求 CPU 时间（wall clock 不计）默认 30s，可调高但非无限。SSE 流式期间 CPU 时间主要用于 chunk 解析与重组，长会话需确认不触发 CPU 限制。
- heartbeat 不能用 `setInterval`（Workers 无持久定时器，DO 内可用 `alarm`）。方案 §4.5 提到 heartbeat 但未说明实现机制。

### 4.3 Durable Object 单 region 与可用性

- DO 实例运行在某个 region 的某台机器上，自动迁移但有冷启动延迟。方案未说明 DO 的 region 策略。
- 若作者与 DeepSeek 端点（目前主要在亚洲）网络延迟敏感，DO 部署 region 会影响 TTFT。方案 §6 要求 TTFT SLO 但未提 region 选择。
- DO 故障域：单 DO 实例损坏即整条会话不可用。第一轮已提恢复演练，本轮补充：**个人项目是否接受"会话丢失即重新开始"**？若接受，可大幅简化状态层。

### 4.4 SQLite DO 的实际限制

Cloudflare SQLite DO 有单 DO 存储上限与单行大小约束。方案 §4.2 持久化 `reasoning_content` + 完整 messages 快照 + 工具 arguments，长会话可能触顶。第一轮提过容量硬限制，本轮补充：**应明确单 conversation 最大 messages 数（如 200）与单 message 最大字节（如 256KB）**，并在 §4.1 request adapter 入口校验，超限返回 413 而非让 SQLite 隐式失败。

## 5. 替代架构未被严肃比较

虽然 §2.1 已确认协议转换不可避免，但"用 Cloudflare Worker + DO"这条路线本身未被与替代方案对比。

### 5.1 最简透传代理（无状态）

- 一个无状态 Worker，把 Responses 请求转 Chat Completions，结果转回 Responses，不存任何状态。
- 代价：不支持 thinking + tools（需回传 reasoning_content）、不支持 previous_response_id。
- 收益：1 天可上线，验证 Codex 基础对话。

**挑战：** 这正是 §1.3 建议的 MVP-A。方案跳过它直接做 stateful，违背 `AGENTS.md` 的"Simplicity First"。

### 5.2 非 Cloudflare 方案

- 本地 Node.js/Bun 进程 + SQLite，作为 Codex 的 `base_url` 指向 `localhost`。
- 代价：需要本机常驻、无公网 TLS（Codex 是否接受自签证书未知）。
- 收益：无平台限制、无 subrequest/CPU 约束、无 DO 计费、调试简单。

**挑战：** 对于"内部小项目"，本地进程可能比 Worker 更合适。方案未给出选择 Worker 的理由（是否需要公网访问？多设备共享？）。若仅个人单机用，本地方案的交付速度更快。

### 5.3 OpenAI 官方 API 作为基线对照

方案 §6 的 golden fixtures 应来自真实 Codex 回放，但录制 OpenAI 真实流量可能涉及其 API ToS。**挑战：** 是否可用 OpenAI 官方 Responses API 作为"参考实现"对照——即同一 Codex 请求分别发给 OpenAI 与本适配器，比对事件流差异？这能替代部分 fixture，且无 ToS 风险（正常 API 调用）。方案未提此对照路径。

## 6. 凭证、租户与额度模型缺失

方案 §4.1 提到 Bearer 鉴权，但未回答：

| 问题 | 个人项目答案 | 多用户答案 | 方案当前 |
|---|---|---|---|
| 谁持有 `CODEX_GATEWAY_API_KEY` | 作者本人 | 每用户独立 | 未定义 |
| DeepSeek API key 归属 | 作者的 | 平台垫付或用户自带 | 未定义 |
| 额度超支谁负责 | 无所谓 | 需按用户计费 | 未定义 |
| key 轮换 | 手动改 secret | 需无停机轮换 | §6 提及但无方案 |
| key 泄露半径 | 单人 | 全平台 | 未评估 |

**挑战：** 即便是个人项目，也应明确"DeepSeek key 存放位置（Worker secret? 环境变量?）、泄露后如何吊销、是否在日志中过滤"。当前方案的可观测性章节（§4.1）只说"不记录 secret"，但未说 secret 如何注入与轮换。

## 7. fixture 采集的可行性与合规性

第一轮要求补 fixture 目录结构，本轮质疑采集本身：

- **合规性：** 录制 Codex 与 OpenAI 之间的真实流量（含 prompt、源码、工具输出）可能违反 OpenAI API ToS 的数据使用条款。方案 §6 要求 fixture 脱敏，但脱敏后的 fixture 是否还能验证"工具输出长文本"等场景？
- **可行性：** 个人项目是否有足够多样的真实任务来覆盖 §5 所列场景（双并行工具、工具失败、连续三轮、断流）？若靠人工构造，本质等同于写测试用例，不如直接写集成测试。
- **替代：** 用 Codex 的 `--dry-run` 或日志模式（若存在）输出请求体，而非录制网络流量。方案未调研此路径。

**挑战：** Phase -1 应先调研 Codex 是否提供"导出请求体"能力，再决定 fixture 采集方式，而非假设必须录制流量。

## 8. 可逆性与迁移成本

方案选定 DO + SQLite 后，未讨论退出路径：

- DO 数据如何导出（`export()` API？SQL dump？）；
- 若 Cloudflare 计费或限制变化，能否迁移到 D1/外部 SQLite；
- schema 变更如何灰度（DO 无标准迁移工具，需手动 `sql` 语句）；
- 若整个 CF 方案失败，能否切到本地进程且保留会话历史。

**挑战：** §6 上线门槛应增加一条："DO 数据可导出为标准 SQLite 文件，且存在非 CF 备选实现路径"。否则一旦绑定，个人项目会被平台锁定。

---

## 9. "直连基线"的可归因性问题

方案 §5 把"直连 DeepSeek"作为 Phase 3 Gateway 对比的基线。但直连时延迟由 Worker 处理 + Worker→DeepSeek 网络两部分组成，无独立测量。若不拆分，Phase 3 无法判断 Gateway 的收益是来自缓存、路由还是纯粹网络差异。

**挑战：** §4.1 observability 模块应记录"Worker 内处理耗时"与"上游 fetch 耗时"两个独立指标，从 Phase 0 起采集，而非等到 Phase 3 才补。

## 10. 对 AGENTS.md 四原则的逐条对照

用项目自己的原则审视方案，是最直接的挑战：

| 原则 | 方案是否符合 | 具体冲突 |
|---|---|---|
| **Think Before Coding** | 符合 | 方案充分思考，未假设。但 §1 的定位未思考清楚（见 §1.3） |
| **Simplicity First** | **违反** | "Minimum code that solves the problem" —— 方案是"最大正确性"，不是"最小代码"。§1.2 三处过度投资直接违反"No abstractions for single-use code" |
| **Surgical Changes** | N/A | 全新项目无既有代码，但若后续把方案套到现有仓库，应警惕一次性引入过多 |
| **Goal-Driven Execution** | **部分违反** | "Define success criteria. Loop until verified." —— §6 给了成功标准，但 §3.1 的 v1 目标与 §5 分阶段未对齐（第一轮 §4.2 已指出），导致"何时算成功"模糊 |

**挑战结论：** 方案最该砍的一刀来自项目自己的 `AGENTS.md`：**"Would a senior engineer say this is overcomplicated? If yes, simplify."** 一个内部小项目上 DO + 四阶段 + SLO 矩阵，大概率会被资深工程师判定为过度设计。

## 11. 第二轮挑战结论与建议

### 11.1 与第一轮的关系

第一轮的结论是"方向批准，实现暂缓，补四份子规格"。第二轮的结论是"在补规格之前，先回答两个更前置的问题"：

1. **这是给谁用的？**（定位）
2. **最小可运行链路是否通？**（可行性）

这两个问题不解决，补多少子规格都是在过度设计的骨架上填肉。

### 11.2 建议的修订动作（按优先级）

| 优先级 | 动作 | 理由 |
|---|---|---|
| P0 | 在方案开头补"项目定位"一节，明确单用户 vs 多用户 | 决定后续所有设计的复杂度上限 |
| P0 | Phase -1 增加"hello-codex"端到端冒烟（最简无状态 Worker） | 验证协议转换链路本身通，先于 fixture |
| P0 | §4.4 增加 effort → DeepSeek 模型名映射，验证 v4-pro 是否支持 `thinking.type=disabled` | 否则 `effort=none` 会系统性 400 |
| P1 | §1.2 三处过度投资按定位砍减 | 落实 `AGENTS.md` Simplicity First |
| P1 | §4 增加 Cloudflare 平台限制小节（subrequest、CPU、DO region、SQLite 上限） | 让实现者知道边界 |
| P1 | Phase -1 增加 DeepSeek 端点能力探测脚本，输出 `deepseek-capabilities.md` | 给"验证后透传"提供验证对象 |
| P2 | §6 增加"DO 数据可导出"与备选实现路径 | 避免平台锁定 |
| P2 | §4.1 拆分"Worker 内处理"与"上游 fetch"两个耗时指标 | 让 Phase 3 基线可归因 |

### 11.3 一句话收尾

> 第一轮让方案"更完整"，第二轮要求方案"先更小"。在项目定位明确、最小链路跑通、模型名映射验证之前，不应启动 Phase 0 的 stateful 实现。
