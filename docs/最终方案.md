# DeepSeek 接入 Codex：最终实施方案

日期：2026-06-19

## 1. 最终决策

采用 **Cloudflare Worker 原生 TypeScript 适配器 + Durable Object SQLite 状态存储**：

```text
Codex Desktop / CLI
  -> Cloudflare Worker（鉴权、限流、Responses 协议适配）
       |-> Conversation Durable Object + SQLite（状态依赖）
       `-> DeepSeek Chat Completions（第一阶段直连）
           或 Cloudflare AI Gateway -> DeepSeek（稳定后按需切换）
```

不采用无状态工具回合：真实 Codex 请求是否携带完整历史尚未证明，DeepSeek thinking 工具回合还要求保存并回传 `reasoning_content`。也不把 AI Gateway 放入第一条可用链路；先直连 DeepSeek 建立协议和延迟基线，之后再用同一测试集验证 Gateway 的收益与副作用。

这是一个 **Worker-first、stateful、contract-first** 的单一路线方案。

## 2. 决策依据

所有文档一致支持以下事实：

1. Codex 自定义 provider 使用 Responses API，DeepSeek provider-native 接口使用 Chat Completions，因此协议转换层不可省略。
2. Worker 原生实现减少运行时与供应链层级，但不能消除协议复杂度。
3. Codex 工具调用是客户端执行，适配器不应执行工具；适配器仍需正确保存和恢复 assistant tool-call turn。
4. DeepSeek thinking + tool calling 要求后续请求回传原 assistant 回合的 `reasoning_content`，这使状态设计成为 Tier 2 的前置条件。
5. Responses item `id` 与函数 `call_id` 是两个标识；SSE 事件生命周期、最终 `response.output` 和错误状态必须完整一致。

因此，最终选择不是“最少代码”，而是“最短链路下可被契约测试证明正确的实现”。

## 3. 支持范围

### 3.1 第一版最终目标

- `POST /v1/responses`，优先支持 `stream=true`
- `GET /v1/models`
- `/healthz` 仅表示 Worker 存活；`/readyz` 验证状态存储与上游配置
- string input 和文本 message input
- `instructions`，按明确优先级映射为 DeepSeek 支持的 system message
- function tools、`tool_choice`；`parallel_tool_calls` 仅在目标模型与上游接口已验证支持时透传，否则明确拒绝
- 单工具和并行工具调用
- `function_call_output` 与原 `call_id` 的严格关联
- `previous_response_id` 状态恢复
- thinking 纯文本，以及通过单独准入测试后的 thinking + tools
- 客户端取消向上游传播
- 上游 4xx、5xx、限流、超时、断流的结构化错误映射

### 3.2 第一版不做

- 服务端工具执行或服务端 tool loop
- Hosted tools、MCP server execution、WebSocket
- 多模态输入
- 任意 OpenAI Responses 全量兼容
- 流开始后自动重试；未证明幂等前，不重放可能已产生工具调用的请求

不支持的字段或 item 必须返回明确的 4xx，禁止静默丢弃、伪造 `unknown_tool` 或返回 Chat Completions 对象冒充 Responses。

## 4. 核心设计

### 4.1 组件

| 模块 | 职责 |
|---|---|
| Worker router | 路由、方法白名单、Bearer 鉴权、请求体上限、速率和并发保护 |
| request adapter | Responses schema 校验；输入、tools、reasoning 到 Chat Completions 的显式白名单映射；Responses function tool 的扁平字段转换为 Chat tool 的 `function` 对象 |
| stream adapter | 解析 DeepSeek SSE，生成完整 Responses 事件序列，维护输出不变量 |
| response adapter | 组装非流式 Responses；若尚未实现则明确返回 501 |
| Conversation Durable Object | 按 conversation key 串行化写入，保存 response、assistant tool call、call id 和 reasoning 上下文 |
| upstream client | 直连/Gateway URL、首字节和 chunk idle timeout、取消传播、错误过滤 |
| observability | request id、状态码、阶段耗时、模型、usage；不记录 prompt、工具输出或 reasoning 正文 |

### 4.2 状态模型

每个会话固定路由到一个 Durable Object。首个请求生成不可预测的 conversation key；适配器签发的 opaque response id 必须能经服务端校验后恢复该 key 和 response 序号，使 `previous_response_id` 无需全局扫描即可回到同一对象。禁止直接信任客户端提供的对象 id。

Durable Object 使用 SQLite，至少保存：

| 数据 | 用途 |
|---|---|
| response id、parent response id、状态 | 恢复 `previous_response_id` 链和终态 |
| 原始 instructions 的规范化结果 | 新回合不继承时按协议处理，不靠猜测 |
| Chat messages 的规范化快照 | 重建 DeepSeek 所需完整历史 |
| Responses item id | 标识输出 item |
| DeepSeek tool call id / Responses call id | 关联 `function_call_output`，并转换为上游 tool message 的 `tool_call_id`；与 item id 分离 |
| tool name、完整 arguments、顺序 | 重建 assistant `tool_calls`，支持并行和交错分片 |
| assistant `reasoning_content` | thinking 工具回合下一请求必须原样回传 |
| created/expires 时间 | TTL 清理和容量控制 |

每个 response 的状态转换采用单写者语义。创建记录、校验 parent、消费 tool output 和进入 `in_progress` 必须在同一事务内完成；重复的 `function_call_output.call_id` 必须被识别并拒绝或幂等处理。只有 `completed` response 可作为下一轮 parent；失败、中断和过期状态返回明确错误。

### 4.3 Responses SSE 契约

文本输出至少满足：

```text
response.created
response.in_progress
response.output_item.added
response.content_part.added
response.output_text.delta *
response.output_text.done
response.content_part.done
response.output_item.done
response.completed
```

函数调用也必须有对应的 `output_item.added/done` 和 `function_call_arguments.delta/done`。每个事件发送 `event: <type>` 与 `data: {...}`，维护单调事件序号。

必须满足以下不变量：

- 每个 added 都有且只有一个 done。
- delta 拼接值等于 done 和最终 `response.output` 中的值。
- `output_index` 连续且不冲突。
- item id 与 call id 分离且跨回合稳定。
- `length`、过滤、上游错误和断流不得伪装为 `completed`。

### 4.4 Reasoning 策略

分阶段启用：

| Codex effort | DeepSeek 策略 |
|---|---|
| `none` | 显式 `thinking.type=disabled` |
| `low` / `medium` | `thinking.type=enabled` + `reasoning_effort=high`；这是 DeepSeek 官方兼容映射，不宣称保留原 effort 语义 |
| `high` | `thinking.type=enabled` + `reasoning_effort=high` |
| `xhigh` | `thinking.type=enabled` + `reasoning_effort=max` |

在 thinking + tools 准入前，工具阶段强制关闭 thinking。启用后必须捕获、持久化并在后续 assistant tool-call message 中回传完整 `reasoning_content`。原始 reasoning 不得伪装成 `output_text`；只有固定 Codex fixture 证明可接受时才转换为 Responses reasoning item，否则仅在服务端保存并向客户端输出最终答案。

### 4.5 超时、取消和重试

分别配置：

- 首字节超时；
- 上游 chunk idle timeout，每次收到 chunk 后重置；
- 可选总请求上限；
- 客户端断开立即取消 reader 和 upstream fetch。

heartbeat 只保持 Worker 到 Codex 的 SSE 活跃，不能替代上游 idle timeout。流已经输出任何事件后，适配器和 Gateway 均不自动重试。

## 5. 实施顺序

### Phase -1：协议勘测与固定契约

固定 Codex Desktop/CLI 版本，采集脱敏 fixture：

- 纯文本；
- 单工具；
- 双并行工具及参数交错分片；
- 工具失败和长工具输出；
- 同一 thread 连续三轮；
- thinking 纯文本与 thinking + tools；
- 客户端取消和流中断。

先回答“Codex 发全量历史还是只发增量 + `previous_response_id`”，再冻结第一版 schema。没有 fixture，不进入 Tier 2 实现。

### Phase 0：严格文本流

实现 Worker 路由、鉴权、请求校验、直连 DeepSeek、正式 SSE 生命周期、错误映射和取消传播。tools、`previous_response_id`、非文本 item 暂时明确拒绝。

### Phase 1：工具调用，不带 thinking

加入 Durable Object、tool schema 映射、item id/call id、并行工具、`function_call_output` 和跨请求历史恢复。此阶段显式关闭 thinking。

### Phase 2：thinking + tools

加入 effort 映射、reasoning 事件策略、`reasoning_content` 保存和回传。连续两次工具调用、多工具和断流用例全部通过后才默认启用。

### Phase 3：Gateway 与生产硬化

在直连基线稳定后加入 AI Gateway，对比 TTFT、总耗时、SSE 中断率、错误码和日志内容。只有收益明确且日志策略可接受时才设为默认。

同时完成限流、额度保护、密钥轮换、TTL 清理、预算告警、结构化指标和回滚流程。

## 6. 上线门槛

以下条件全部通过才可称为可日常使用：

- 固定 Codex 版本的所有 golden fixtures 通过。
- 文本、单工具、并行工具、工具失败、连续工具回合均端到端成功。
- thinking + tools 不丢 `reasoning_content`，上游无 400。
- `previous_response_id` 可在 isolate 重启后恢复；过期时返回明确错误。
- 所有 SSE 生命周期和最终 output 不变量通过属性/fixture 测试。
- 上游 401、429、5xx、首字节超时、idle timeout、坏 chunk 和断流均不伪装为成功。
- 客户端取消后上游调用及时结束，不继续消耗额度。
- 2 并发和 6 并发下有 TTFT、内存、错误率数据，并达到预设 SLO。
- 日志不包含 prompt、源代码、工具输出、reasoning 正文或 secret。
- 依赖和 compatibility date 固定，lockfile 提交，可回滚。

## 7. 失败与幂等边界

- 客户端可提供 request id；同一会话内重复 request id 必须返回原结果或明确冲突，不得再次调用上游。
- 首个 SSE 事件发出前，仅对确定未到达上游的连接错误做有界重试。
- 首个 SSE 事件发出后不自动重试；断流将 response 标记为 `failed` 或 `incomplete`，不得写成 `completed`。
- 工具执行发生在 Codex 客户端。适配器只转换与保存工具调用，不创建服务端 tool loop。
- `function_call_output` 只能消费一次，且 tool name、call id、parent response 必须同时匹配。
- SQLite 写入失败、上游取消和客户端取消都必须落到可诊断终态，并记录不含正文的 request id 与阶段信息。

## 8. 配置原则

Codex 保持：

```toml
model = "deepseek-v4-pro"
model_provider = "cf_deepseek_adapter"

[model_providers.cf_deepseek_adapter]
name = "Cloudflare DeepSeek Responses Adapter"
base_url = "https://codex-adapter.example.com/v1"
env_key = "CODEX_GATEWAY_API_KEY"
wire_api = "responses"
stream_idle_timeout_ms = 300000
supports_websockets = false
```

在断流幂等性验证完成前，不主动提高 `stream_max_retries`。模型名、upstream mode、reasoning 映射和超时集中在服务端配置，不散落到多个 agent 文件。
