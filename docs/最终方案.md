# DeepSeek 接入 Codex：最终实施方案

日期：2026-06-19
状态：经两轮挑战评审修订，作为当前唯一活动方案

## 1. 项目定位

本项目是 **单维护者、单用户、少量自用设备** 的内部工具，目标是在 Cloudflare Worker 上尽快跑通 Codex 到 DeepSeek 的日常文本与工具调用，而不是建设多租户代理服务。

当前边界：

- 一个共享的适配器访问密钥，一个维护者持有的 DeepSeek API key；
- 低并发，不承诺公网 SaaS 的可用性、计费隔离或租户隔离；
- 会话状态丢失时允许新开会话，不为历史迁移和跨区容灾提前建设系统；
- Cloudflare 是目标部署环境；本地 Worker 仅用于开发和对照，不另建 Node.js/Bun 产品路线；
- 优先交付最小可用链路，只有实测证明必要时才增加状态层。

若未来开放给其他用户，必须另开设计评审，补齐多租户鉴权、配额、隐私、审计、SLO 和故障恢复；本方案不能直接外推为公共服务规格。

## 2. 最终决策

采用 **Cloudflare Worker 原生 TypeScript、直连 DeepSeek、按证据升级状态能力** 的路线：

```text
MVP-A
Codex Desktop / CLI
  -> Cloudflare Worker（鉴权、Responses -> Chat Completions、SSE 转换）
       `-> DeepSeek Chat Completions

仅当工具回合或 previous_response_id 的实测证明必须保存服务端状态时
  -> Conversation Durable Object + SQLite
```

关键决策：

1. **协议适配不可省略。** Codex 自定义 provider 当前只支持 `wire_api = "responses"`，DeepSeek 原生接口使用 Chat Completions。
2. **MVP 无状态。** 第一版只做流式纯文本，不预先引入 Durable Object、KV、全局 registry 或会话迁移。
3. **状态层是条件能力。** Phase 1 的真实 fixture 若证明 Codex 输入不足以重建 DeepSeek 工具回合，再直接引入 Durable Object；不先上 KV 作为过渡。
4. **Gateway 不在活动路线。** 先建立直连的正确性和延迟基线；除非直连出现可量化问题，否则不实施 AI Gateway。
5. **不预先追求完整 Responses 兼容。** 只实现固定 Codex 版本实际使用且已有 fixture 的字段和事件。

这是一个 **Worker-first、MVP-first、evidence-driven** 的方案，不再把 stateful 视为所有阶段的默认前提。

## 3. 交付范围

### 3.1 MVP-A：流式纯文本

首个可部署版本只支持：

- `POST /v1/responses`，仅 `stream=true`；
- `GET /v1/models`，仅返回适配器实际允许的模型别名；
- `GET /healthz`，只表示 Worker 进程存活；
- string input、文本 message input 和 `instructions`；
- 纯文本请求中的 `reasoning.effort`，按 Phase -1 已验证的模型与 thinking 参数映射；
- Bearer 鉴权、请求体上限、结构化错误、客户端取消传播；
- DeepSeek 文本流到 Codex 可接受的最小 Responses SSE 事件序列。

MVP-A 明确拒绝：

- tools、`function_call_output`、`previous_response_id`；
- `stream=false`、reasoning item、多模态和其他 Responses item；
- Hosted tools、MCP server execution、WebSocket；
- 自动重试已经到达上游或已经开始输出的请求。

不支持的字段返回明确的 4xx；尚未实现的传输模式返回明确的 501。禁止静默丢弃字段、伪造工具或返回 Chat Completions 对象冒充 Responses。

### 3.2 后续能力

| 能力 | 进入条件 | 交付定义 |
|---|---|---|
| MVP-B：工具调用，不带 thinking | MVP-A 稳定且 Codex 工具 fixture 已冻结 | 单工具端到端可用；并行工具仅在端点探测通过后加入 |
| MVP-C：thinking + tools | MVP-B 稳定，reasoning 对 Codex 的展示策略和 DeepSeek 回传规则已验证 | 连续工具回合不丢 `reasoning_content` |
| previous_response_id | fixture 证明 Codex 依赖增量恢复 | isolate 重启后可恢复；过期和冲突有明确错误 |
| AI Gateway | 直连存在可量化的可靠性、观测或成本问题 | 用相同负载对比后收益明确 |

后续能力不是 MVP-A 的隐藏验收项，也不预先排期。

## 4. Phase -1：先证明链路

Phase -1 是编码前的短周期可行性勘测，但其第一目标是跑通链路，不是先写完整子规格。

### 4.1 必须完成的实测

1. 固定 Codex CLI/Desktop 的准确版本并记录采集命令。
2. 编写最简无状态 Worker，让 Codex 完成一次 `hello` 纯文本流式对话。
3. 记录 `examples/hello-codex.md`：版本、配置、执行命令、脱敏后的请求形态、结果和已知限制。
4. 直接探测 DeepSeek 端点并生成 `docs/deepseek-capabilities.md`，至少覆盖：
   - 当前允许的模型名；
   - `thinking.type=disabled/enabled`；
   - `reasoning_effort=high/max`；
   - `tool_choice` 的实际支持值；
   - 单工具、多个 tool calls、thinking + tools；
   - `strict` 是否要求 beta endpoint。
5. 确认 Codex 是否总是流式、是否发送全量历史、是否依赖 `previous_response_id`、是否默认发送 `parallel_tool_calls` 或 `reasoning`。

任何能力都以 **当前端点实测** 为准，不能仅凭文档存在某个字段就宣布支持。

### 4.2 fixture 约定

Phase -1 至少建立：

```text
fixtures/
  codex/<exact-version>/
    README.md
    text-request.json
    text-events.jsonl
  deepseek/<date>/
    capabilities.json
```

`README.md` 必须写明采集命令、运行表面（CLI 或 Desktop）、脱敏规则和通过标准。fixture 只使用专门构造的合成 prompt，不录制真实仓库源码、密钥、用户路径或生产工具输出；长文本用合成数据覆盖。

不要求中间人录制 Codex 与 OpenAI 的私有流量。若需参照，只通过正常调用官方 Responses API 获取同一合成请求的结果，并把它作为行为参考，而不是要求字节级复制。

### 4.3 Phase -1 出口

以下条件全部满足才进入 MVP-A：

- `hello-codex` 从真实 Codex 客户端端到端成功；
- 文本请求和事件 fixture 可离线回放；
- DeepSeek 模型与 thinking 开关的探测结果已落盘；
- 没有发现必须在首版引入状态层的硬阻塞。

若最小链路不通，停止并记录阻塞，不继续设计 DO 或工具协议。

## 5. MVP-A 设计

### 5.1 最小组件

| 模块 | 职责 |
|---|---|
| router | 路由、方法白名单、Bearer 鉴权、请求体上限 |
| request adapter | 仅接受冻结 fixture 中的文本字段，显式映射到 Chat Completions |
| upstream client | 直连 DeepSeek、模型白名单、超时和取消 |
| stream adapter | 解析 DeepSeek SSE，生成 Codex 实测需要的 Responses 事件 |
| observability | request id、状态码、Worker 处理耗时、上游等待耗时、usage；不记录正文 |

MVP-A 不包含 response adapter、状态仓库、`/readyz`、服务端工具循环或通用 Responses schema 框架。

### 5.2 模型选择

Codex 只看到稳定的服务端别名，例如 `deepseek-codex`。Worker 配置维护显式白名单和别名映射，不直接信任请求中的任意模型名。

Phase -1 前不假定 `deepseek-v4-pro`、`deepseek-v4-flash` 与 thinking 模式的固定关系。初始规则是：

- MVP-A 根据探测结果把 `none` 映射到已验证的非 thinking 配置；
- MVP-A 可把 `low`、`medium`、`high`、`xhigh` 映射到已验证的 thinking 模型与 effort，但只输出最终文本，不生成 Responses reasoning item；
- 若 Codex 依赖 reasoning item 才能完成纯文本请求，Phase -1 判定 MVP-A 设计不成立并先修订契约；
- MVP-C 再扩展并冻结适用于工具回合的 `effort -> upstream model + thinking + reasoning_effort` 映射；
- 同一状态会话是否允许跨模型续接，必须由 thinking + tools 探测决定，默认不允许。

DeepSeek 当前文档列出 `deepseek-v4-flash`、`deepseek-v4-pro`，并注明旧模型名将在 2026-07-24 15:59 UTC 弃用。模型名属于运行配置和能力探测结果，不散落在适配逻辑中。

### 5.3 SSE 契约

MVP-A 不从文档猜测一套“完整事件集”，而是冻结真实 Codex 已接受的最小文本事件流。目标序列从以下候选开始，以 fixture 为最终准绳：

```text
response.created
response.in_progress
response.output_item.added
response.content_part.added
response.output_text.delta *
response.output_text.done
response.content_part.done
response.output_item.done
response.completed
```

每个事件使用 `event: <type>` 与 `data: {...}`。是否需要 `sequence_number`、具体必填字段和错误终态全部由 hello 实测与 golden fixture 冻结。

必须保持：

- added/done 成对；
- delta 拼接值等于 done 和最终 output；
- `output_index` 连续；
- 上游过滤、错误和断流不得伪装成 `completed`。

### 5.4 超时、取消和重试

- 分开设置首字节超时、上游 chunk idle timeout 和可选总时限；
- 将客户端 `request.signal` 与内部 `AbortController` 连接，取消上游 fetch，并显式取消未消费的 response body；
- 用集成测试验证客户端断开后上游在规定时间内结束，不能只验证本地 reader 停止；
- MVP-A 默认不发送 heartbeat；只有实测出现中间链路 idle 问题时才加入；
- 首个 SSE 事件发出后不自动重试；在未证明安全前，Codex provider 的 `stream_max_retries` 设为 0。

### 5.5 凭证、日志和限制

- 适配器访问密钥通过 Codex 环境变量注入；
- DeepSeek API key 使用 Cloudflare Worker secret，禁止放入仓库、普通变量或响应；
- 轮换方式是创建新 secret、部署验证、撤销旧 key，单用户阶段不建设无停机多 key 系统；
- 日志只保留 request id、阶段、耗时、状态和 usage，不记录 prompt、tool output、reasoning、Authorization 或上游响应正文；
- MVP-A 请求体上限先设为 1 MiB，超限返回 413；Phase -1 若证明正常 Codex 请求超过该值，再按 fixture 调整。

Cloudflare 的 CPU、子请求、并发连接和响应大小限制以实施当日官方文档为准。当前方案不把平台最大值当作应用目标；长流式请求必须通过 `wrangler dev` 与部署环境各验证一次。

## 6. 条件状态设计

Durable Object 不是 MVP-A 的依赖。只有以下任一证据出现时才启用：

- Codex 只发送增量 input，DeepSeek 需要服务端恢复完整工具历史；
- thinking + tools 必须跨请求原样回传服务端才拥有的 `reasoning_content`；
- `previous_response_id` 是目标 Codex 流程的硬需求。

启用前必须冻结四项子规格：

1. `docs/response-id.md`：ID 格式、签名、过期、密钥轮换和错误码；
2. `docs/conversation-schema.sql`：表、索引、状态机和事务边界；
3. `docs/sse-golden/`：文本、单工具、并行工具和 reasoning 的事件 fixture；
4. `docs/threat-model.md`：持久化正文、删除、导出、调试和泄露半径。

预选但未实施的 response id 方案为签名 token：不可预测 conversation id + response 序号 + 签发/过期时间 + key id，经 HMAC 校验后路由到 `idFromName(conversation_id)`。最终格式须以 Codex fixture 为依据。畸形或签名失败返回 400，已过期或不存在返回 404，parent 状态冲突返回 409。

状态层启用时还必须明确：

- 全量 input 与增量恢复二选一，删除另一条重建路径；
- assistant 的多个 tool calls 在一个 message 中聚合；item id 与 call id 分离；
- 单 message、单 conversation、parent 深度和 TTL 的应用级硬限制；
- `in_progress` 崩溃恢复、tool output 单次消费和事务状态机；
- 标准 JSON 导出与 `deleteAll()` 删除路径。个人项目不要求导出 Cloudflare 内部 SQLite 文件。

不使用 KV 保存工具状态或 reasoning：它不能提供该路径需要的同一会话强一致事务，且会制造额外迁移。

## 7. 实施顺序与验收

### Phase -1：可行性勘测

交付 hello、Codex 文本 fixture 和 DeepSeek 能力矩阵。预计 1–3 天；未通过则停止。

### Phase 0：MVP-A

建立最小 Worker 工程、锁定依赖和 compatibility date，实现严格文本流、鉴权、错误、超时、取消和日志。

通过标准：

- 固定 Codex 版本完成 30 次连续合成文本请求，适配器无 5xx、无伪 completed；
- 文本 SSE golden fixture 全部通过；
- 上游 401、429、5xx、首字节超时、idle timeout、坏 chunk 和中途断流均映射为失败；
- 客户端取消后上游调用及时结束；
- 日志抽查不含请求正文、响应正文或密钥；
- 记录 TTFT 和端到端耗时基线，但暂不设置脱离上游基线的硬 SLO。

### Phase 1：MVP-B

先冻结单工具 fixture，再实现工具调用。只有实际需要时才加入 DO。并行工具、strict 和 `previous_response_id` 各自通过端点与 Codex fixture 后独立启用，不能捆绑宣布完成。

### Phase 2：MVP-C

实现 reasoning 暴露策略、effort 映射及 `reasoning_content` 的字节级保存与回传。连续两次工具调用、并行工具和断流用例全部通过后才默认启用。

没有 Phase 3。Gateway、多租户、并发 SLO、跨区恢复和平台迁移保留在 backlog，只有出现明确需求或数据时再立项。

## 8. 配置草案

配置键必须用 Phase -1 的真实 Codex 版本验证：

```toml
model = "deepseek-codex"
model_provider = "cf_deepseek_adapter"

[model_providers.cf_deepseek_adapter]
name = "Cloudflare DeepSeek Responses Adapter"
base_url = "https://codex-adapter.example.com/v1"
env_key = "CODEX_GATEWAY_API_KEY"
wire_api = "responses"
stream_idle_timeout_ms = 300000
stream_max_retries = 0
supports_websockets = false
```

provider 配置放在用户级 Codex 配置中；服务端只允许 `deepseek-codex` 别名，并由单一配置决定真实 DeepSeek 模型。

## 9. 对挑战报告的处理结论

| 挑战 | 处理 |
|---|---|
| 项目体量与内部小项目冲突 | 接受；明确单用户定位，删去活动路线中的 Gateway、多租户和完整生产 SLO |
| 先证明最小链路 | 接受；hello-codex 成为 Phase -1 首要出口 |
| fixture、脱敏、golden 不明确 | 接受；定义最小目录、合成数据和回放要求 |
| response id、SQL、威胁模型未冻结 | 接受但后移；作为启用状态层前的门禁，不阻塞无状态 MVP-A |
| DO 默认引入过重 | 接受；DO 改为证据触发；不接受 KV 中间层 |
| effort 与模型名假设 | 接受；先做端点探测，MVP-A 接受纯文本 effort 但不生成 reasoning item，MVP-C 再冻结工具回合映射 |
| Cloudflare 平台限制缺失 | 接受；加入取消、CPU/连接/大小验证和应用级请求上限 |
| 工具、strict、parallel 支持未经验证 | 接受；加入 DeepSeek 能力矩阵，逐项启用 |
| 幂等 request id 协议不明 | 调整；MVP-A 不承诺流重放或幂等缓存，关闭流重试；需要状态时再定义 |
| reasoning 对 Codex 是否硬依赖 | 接受；Phase -1 先验证纯文本是否依赖 reasoning item，依赖则先修订 MVP-A；thinking + tools 仍后置 |
| 本地进程替代 Worker | 不采纳为产品路线；项目目标就是 Cloudflare 部署，本地实现仅用于测试 |
| DO 数据导出与平台退出 | 部分接受；状态启用时提供标准 JSON 导出，不为个人项目建设跨平台会话迁移 |
