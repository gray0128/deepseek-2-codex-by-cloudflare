# 对《DeepSeek 接入 Codex：最终实施方案》的挑战性评审

---
评审日期：2026-06-19
评审对象：`docs/最终方案.md`
评审立场：认可其对早期 Worker 原稿 P0 问题的修正方向；重点挑战仍待证明的假设、未冻结的接口细节、Durable Object 成本与复杂度、以及「可验收」与「可日常用」之间的差距。
---

## 0. 结论

**最终方案比早期原稿成熟一个量级，但仍不能直接进入 Phase 0 编码。**

它正确吸收了 `docs/archive/适配器方案/方案挑战-codex.md` 的核心教训：放弃默认无状态、补齐 SSE 生命周期、分离 item id 与 call id、分阶段处理 thinking + tools、区分 idle timeout 与总超时、直连 DeepSeek 作为基线。这些修正使架构方向从「乐观 Tier 2」变为「可被契约证明的正确性优先」。

然而，方案在 **Codex 真实请求形态、opaque response id 设计、SQLite 数据模型、reasoning 对 Codex 的暴露策略、SLO 数值、运维与成本** 上仍留有大量空白。其中若干项不是实现细节，而是会改变模块边界的前置决策。

**评审决策：**

| 决策项 | 结论 |
|---|---|
| Worker + DO SQLite 主路线 | 保留，比无状态或纯 KV 更贴近 DeepSeek thinking + tools 约束 |
| Phase -1 前置 | 完全同意；且应视为 **硬门槛**，不是可并行跳过的文档工作 |
| Phase 0 立即开工 | **不接受**；至少缺 fixture、response id 编码、SQL schema、SSE golden 事件集 |
| 「第一版最终目标」范围 | 方向正确，但把 thinking 纯文本、并行工具、previous_response_id、取消传播同时列为 v1，工期与风险被低估 |
| 作为唯一活动方案 | 可保留，但需在本报告所列 P0 项冻结后增补 **附录：可执行规格** |

> 建议状态：从「可实施规格」降级为 **「方向正确的实施纲领 + 待冻结子规格」**。
> 预计：Phase -1 完成且 P0 设计冻结后，Phase 0–1 仍需约 2–3 周全职当量；Phase 2 thinking + tools 另计，且高度依赖 Codex fixture。

## 1. 最终方案已修正、不应重复争论的点

以下问题在早期挑战中已充分论证，最终方案的处理 **方向正确**，后续实现不应回退：

1. **状态不可省略**：DeepSeek thinking + tool call 要求回传 `reasoning_content`；`function_call_output` 也可能缺少完整 assistant turn。
2. **禁止静默丢弃与伪造工具**：明确 4xx，禁止 `unknown_tool`。
3. **正式 SSE 生命周期**：added/done 配对、delta 与最终 output 一致。
4. **item id ≠ call id**：跨回合关联必须分字段保存。
5. **超时语义拆分**：首字节、chunk idle、总上限、客户端取消传播。
6. **Gateway 后置**：先建立直连协议基线。
7. **流开始后不重试**：在未证明幂等前合理。

本报告 **不重复** 上述论证，只挑战最终方案 **尚未闭合** 的部分。

## 2. P0 阻断项：不冻结则不能写代码

### 2.1 Phase -1 描述正确，但缺少可执行交付物

方案要求先回答「Codex 发全量历史还是增量 + `previous_response_id`」，并采集 fixture。这是正确门槛，但文档未定义：

| 缺失项 | 为何阻断 |
|---|---|
| fixture 目录结构与命名 | 无 golden 集则 SSE/请求适配无法回归 |
| 固定 Codex 版本号与采集命令 | Desktop/CLI 行为可能分叉 |
| 脱敏规则 | 工具输出常含路径、源码、密钥片段 |
| 通过/失败判定脚本 | 「采集了」不等于「可验收」 |
| 对官方 OpenAI endpoint 的参照录制 | 只有第三方 OpenResponses 仍不足以证明 Codex 容忍度 |

**挑战结论：** 在 `fixtures/codex/<version>/` 存在且 CI 可回放前，任何「Tier 1 原型」都只是猜测。建议把 Phase -1 的完成定义写成检查清单，而不是原则性描述。

### 2.2 opaque response id 与 conversation 路由未设计

§4.2 要求：

- 首个请求生成不可预测 conversation key；
- response id 可校验并恢复 key 与序号；
- 禁止信任客户端对象 id。

但方案 **没有** 给出：

- id 编码格式（前缀、版本位、HMAC、过期嵌入）；
- 校验密钥轮换策略；
- conversation key 如何从首个 response 绑定到 DO `idFromName()`；
- 客户端篡改 response id 时的错误码；
- 同一用户多 thread 是否共享 DO。

这是 **安全 + 状态恢复** 的交汇点，不是 Phase 1 再补的细枝末节。若实现为裸 `resp_${uuid}` 而无 MAC，攻击者可向任意 DO 注入伪造 parent chain。

**必须冻结其一：**

- 方案 A：签名 opaque token（无 DO 查表即可验签）；
- 方案 B：随机 token + DO 侧 `response_registry` 表；
- 方案 C：显式 `conversation_id` 字段（若 Codex 实际发送）。

在 Phase -1 fixture 证明 Codex **从不** 发送 conversation 标识前，不要假设仅靠 `previous_response_id` 即可路由。

### 2.3 SQLite 数据模型只有字段清单，没有事务边界

§4.2 表格列出了应保存的数据，但未回答实现者每天都会问的问题：

| 问题 | 风险 |
|---|---|
| messages 存 JSON 快照还是规范化多表？ | 影响重建逻辑、迁移、查询性能 |
| 单条 `reasoning_content` 与工具 arguments 上限？ | 128MB isolate + SQLite 行大小 |
| TTL 如何触发？`alarm()`？独立 cron Worker？ | 过期会话堆积，存储计费上升 |
| parent 链深度上限？ | 恶意或长任务导致链遍历成本 |
| `in_progress` 崩溃后如何恢复？ | 客户端重试 vs 新 request id 冲突 |
| 并行工具：一个 response 内多条 assistant message 还是单条聚合？ | DeepSeek 期望后者；Responses 事件如何映射 |

§4.2 提到「单写者语义」和「同一事务内完成 parent 校验与 tool output 消费」，但没有示例 SQL 或状态机图。**没有状态机图，工程师会各自实现互斥语义。**

### 2.4 Responses SSE 契约仍部分依赖「将来证明」

方案列举的事件序列比早期原稿完整，但仍有两处投机：

1. **reasoning 是否作为 Responses item 暴露给 Codex** —— 取决于 fixture，却未定义失败时的产品行为（隐藏 / 摘要 / 固定占位 / 明确 501）。
2. **`event:` 行与 `sequence_number`** —— 方案要求 `event: <type>`，但未说明是否必须携带 OpenAI 文档中的序列号字段；Codex 解析器是否依赖未知。

此外，DeepSeek thinking 流可能 **先输出 reasoning delta、后输出 content delta**。方案未规定：

- 是否生成 `reasoning_summary_text.delta` 类事件；
- 纯 thinking 无 content 时 `output_text` 生命周期是否仍要走完整套 added/done；
- 工具调用与 reasoning 交错时的 `output_index` 分配规则。

**挑战结论：** 应把「文本 SSE golden 事件」与「工具 SSE golden 事件」拆成两份冻结文件，且分别来自 **真实 Codex 回放**，不是从 OpenAI 文档抄写。

### 2.5 reasoning 策略对 Codex 仍不完整

§4.4 effort 映射与 DeepSeek 文档基本一致，但遗漏了与 Codex 直接相关的语义：

| 遗漏点 | 来源 |
|---|---|
| DeepSeek 对「复杂 agent 请求」可能自动把 effort 提到 `max` | DeepSeek Thinking Mode 文档 |
| 非 tool 回合的 `reasoning_content` 可不回传；tool 回合必须回传 | 同上 |
| 同一线程 Turn 2 仍可能携带 Turn 1 的 reasoning（DeepSeek 样例） | 同上 |
| Phase 1 强制 `thinking.type=disabled` 时，Codex 仍可能发 `reasoning.effort` | Codex 配置默认 high |

方案写「只有 fixture 证明可接受时才转换为 Responses reasoning item」，这是审慎的，但未定义 **Codex 在完全看不到 reasoning 时是否仍可用**。若 Codex UI/日志依赖 reasoning item，则「仅服务端保存」可能导致体验降级，却不触发测试失败。

**必须先回答：** Codex 对 reasoning item 是 **硬依赖** 还是 **可选增强**？这决定 Phase 2 是阻塞项还是锦上添花。

### 2.6 `parallel_tool_calls` 与 `strict` tools 的入口行为未定义

方案对 `parallel_tool_calls` 采取「验证后透传，否则拒绝」，方向正确。但缺少：

- Codex 是否 **总是** 发送该字段；
- 拒绝时返回 400 还是忽略并强制串行；
- DeepSeek `strict` beta（`/beta` base URL、schema 校验）与 Codex 工具 schema 的兼容性策略。

若 Codex 默认 strict tools，而适配器未切换 beta endpoint 或未映射 `strict: true`，会出现 **首轮工具就 400** 的系统性失败，而非边缘 case。

### 2.7 幂等与 `request id` 只有原则，没有协议

§7 要求同会话重复 request id 返回原结果或冲突，但缺失：

- request id 来自 Codex 哪个字段（`metadata`？自定义 header？）；
- 流式响应如何缓存重放（DO 存事件 log？仅存终态？）；
- 冲突定义：同 id 不同 body 是否 409；
- 与 `previous_response_id` 交叉时的优先级。

在流式场景下，「返回原结果」意味着要实现 **SSE 重放或 409 降级**，复杂度不低。若不做，客户端重试会重复消耗 DeepSeek 额度，与 §6「取消后不继续消耗」目标冲突。

## 3. P1 高风险：不阻断开工，但会在 Phase 1–2 爆雷

### 3.1 Durable Object 是正确方向，但成本与热点被低估

选用 SQLite DO 合理，但方案成本章节几乎空白：

- Cloudflare 已于 2025-12 公告 **SQLite 存储计费**（2026 年起），长会话、大工具输出、持久化 `reasoning_content` 都会产生持续存储与读写费用；
- 每个 conversation 串行化：Codex 并行子智能体若共享 thread，会在 DO 入口排队；
- 单 DO 故障域：存储损坏或逻辑 bug 影响整条会话链，PITR 虽可用但方案未写恢复演练。

**挑战：** 应给出「单会话最大消息数 / 最大持久化字节 / 过期时间」三项硬限制，并在超限时返回可诊断 413 或 400，而不是依赖 SQLite 隐式失败。

### 3.2 历史重建 vs 全量 input：双路径未收敛

方案同时假设：

- DO 保存完整 messages 快照；
- 请求适配器做 Responses → Chat 白名单映射。

若 Phase -1 证明 Codex **每次发送全量 input**，则 DO 主要用于 `reasoning_content`、call id 映射与 parent 校验，而非 messages 重建。此时应 **删减** 持久化字段，降低隐私面与存储成本。

若证明 Codex **只发增量**，则 DO 是主路径，request adapter 不能信任 input 中的 assistant turn，必须完全从存储恢复。

**当前方案想同时兼容两种可能，会导致实现过度设计。** 应在 Phase -1 后 **二选一**，并删除另一路径的规格。

### 3.3 `/readyz`、非流式、models 列表仍过于模糊

| 端点 | 挑战 |
|---|---|
| `/readyz` | 未说明探测 DO 的方式（自检 DO？ping SQLite？上游 HEAD？） |
| 非流式 `stream=false` | 明确 501 合理，但未验证 Codex 是否在某些路径强制非流式 |
| `GET /v1/models` | 静态列表是否包含 Codex 配置的 `deepseek-v4-pro` 别名与 DeepSeek 真名映射 |

### 3.4 可观测性与隐私目标存在张力

§4.1 要求不记录 prompt、工具输出、reasoning 正文，但 DO SQLite **必然** 持久化这些内容。方案未定义：

- 存储加密 at rest 是否必要；
- 导出/调试接口如何禁止；
- 合规 retention 与 `deleteAll()` 用户触发删除；
- `wrangler tail` 与异常 dump 的脱敏规则。

「日志不记」不等于「系统不存」。审计时应把 **DO 数据面** 纳入威胁模型。

### 3.5 取消链与 heartbeat 仍缺实现级要求

原则正确（客户端断开 → 取消上游），但未规定：

- `ReadableStream` `cancel()` 是否调用 `gen.return()`；
- heartbeat 与上游 idle timeout 的计时器是否互相干扰；
- 取消后 DO 中 `in_progress` 记录由谁收尾。

建议在规格中增加 **一条必须通过的集成测试**：客户端在首个 tool delta 后断开，断言上游 abort 在 N 秒内发生且 DO 状态为 `failed`/`incomplete`。

### 3.6 上线门槛引用了「预设 SLO」，但方案未给出数值

§6 要求 2 并发与 6 并发下达到预设 SLO，却未定义：

- TTFT P50/P95 上限；
- 端到端错误率；
- 内存峰值；
- SSE 断流率。

没有数字的 SLO 无法验收，也无法在 Phase 3 判断 Gateway 是否值得切换。

### 3.7 Container 回退路径被移除

早期原稿保留 Container/OpenResponses 作为 fallback；最终方案未提及。对于个人项目，在协议高度不确定阶段，**保留切换开关** 能降低路线赌注。这不与 Worker-first 矛盾，属于运维韧性。

## 4. 对实施顺序的具体挑战

### 4.1 Phase 编号合理，但时间预算缺失

| Phase | 挑战 |
|---|---|
| -1 | 1–2 天仅当 Codex 环境已就绪；否则应写 3–5 天 |
| 0 | 若 SSE golden 未冻结，「严格文本流」会返工 |
| 1 | DO + 工具 + previous_response_id 是最大块，与 Phase 2 部分重叠，不应假设线性独立 |
| 2 | thinking + tools 依赖 DeepSeek 子回合语义，测试矩阵应翻倍 |
| 3 | Gateway 对比需要可重复的负载脚本，不是手工 curl |

### 4.2 「第一版最终目标」与分阶段实施存在张力

§3.1 列出的 v1 目标覆盖：流式文本、tools、previous_response_id、thinking、取消、错误映射。
§5 却将 thinking 推到 Phase 2，将 previous_response_id 推到 Phase 1。

**挑战结论：** §3.1 应改名为 **「完整产品目标」**，并增加 **「MVP 切口」** 子节，例如：

- MVP-A：Phase 0 文本流 + Codex 单句；
- MVP-B：Phase 1 工具无 thinking；
- MVP-C：Phase 2 thinking + tools；
- MVP-D：§3.1 全量。

否则读者会误以为第一个可部署版本就要满足 §6 全部门槛。

### 4.3 仓库现状与方案脱节

当前仓库 **无任何实现、无 wrangler、无 tests、无 fixtures**。最终方案未包含最小工程脚手架清单。作为「唯一活动方案」，应至少指向：

- `wrangler.toml` 中 `new_sqlite_classes` 迁移；
- `vitest` + SSE 字节切片测试；
- fixture 目录约定；
- lockfile 策略（早期原稿口喊「锁定」却用 caret，最终方案提到 lockfile 但未写入文件结构）。

## 5. 建议的验收矩阵增补

在最终方案 §6 基础上，建议 **强制新增** 以下用例，否则「可日常使用」判断仍不可靠：

| 类别 | 必测用例 | 通过标准 |
|---|---|---|
| response id | 伪造 id、过期 id、跨 conversation parent | 401/404/409，不污染 DO |
| 历史模式 | 全量 input vs 增量 + previous_response_id | 二选一策略下 DeepSeek messages 与官方样例等价 |
| reasoning | 纯文本多轮、tool 前 reasoning、连续 tool 子回合 | 无 400；存储与回传字节级一致 |
| 并行 | 双 tool 交错参数、完成顺序≠声明顺序 | call id 关联正确 |
| 断流 | 文本中途、tool 参数中途、completed 前 | 状态 `failed`/`incomplete`，不重试上游 |
| 取消 | 首字节前、delta 中、tool 后 | 上游 abort，额度不继续增长 |
| 极限 | 长工具输出、深 parent 链、宽 messages | 超限明确 4xx/413，isolate 不 OOM |
| 重放 | 同 request id 重试 | 不双扣费；行为符合 §7 |
| Codex 回归 | 固定版本：单句、改文件、shell、双工具任务 | 端到端成功 |

每个 `response.completed` 仍应做五项不变量检查（added/done 配对、delta 一致、output_index 连续、id 分离、status 与 finish_reason 一致）——最终方案已写，**应提升到 CI 门禁**。

## 6. 建议在最终方案中增补的规格（最小集合）

在进入 Phase 0 前，向 `docs/最终方案.md` 追加或链接以下子文档即可，不必重写主方案：

1. **`fixtures/codex/<version>/README.md`**：采集方法、脱敏规则、必含场景列表。
2. **`docs/response-id.md`**：编码、签名、过期、错误码。
3. **`docs/conversation-schema.sql`**：表结构、索引、事务状态机。
4. **`docs/sse-golden/*.jsonl`**：文本与工具两套期望事件流。
5. **`docs/slo.md`**：TTFT、错误率、并发、内存的数字门槛。
6. **`docs/threat-model.md`**：DO 存储、日志、Gateway 日志、密钥轮换。

## 7. 最终挑战结论

最终方案已经成功把项目从「低估协议复杂度的 Tier 2 骨架」拉回到 **contract-first、stateful、可阶段性交付** 的正确轨道。它值得作为唯一活动方案。

但它仍然是一份 **纲领**，不是 **可直接分配的工程规格**。当前最大的风险不是 Cloudflare 或 DeepSeek 能力不够，而是：

1. **Codex 真实请求形态仍未被仓库内 fixture 证明**；
2. **状态层关键设计（response id、SQL、事务状态机）尚未落笔**；
3. **reasoning 对 Codex 是硬需求还是软需求尚未定性**；
4. **SLO 与成本边界未数字化的「可日常使用」无法裁判**。

在完成 Phase -1 并冻结上述四项前，推荐结论为：

> **方向批准，实现暂缓。先用 3–5 天完成协议勘测与四份子规格冻结，再启动 Phase 0；Phase 0–1 目标应是「MVP-B：无 thinking 的工具可用」，而非 §3.1 全量。**
